### CM-1 — Policy and Procedures (Control)

This requirement is about making sure your organisation has clear, written rules for how system settings and changes are handled, and that everyone who needs to follow those rules knows what to do. Without this, important changes could be made in different ways, old or incorrect settings might remain in place, and mistakes or unauthorised changes could go unnoticed—raising the risk of service disruption, data loss, and failure to meet legal and regulatory duties.

To meet this expectation, the organisation must create an organisation-wide configuration management policy and supporting procedures. These must explain why they exist, what they cover, who is responsible, how senior leadership is committed, how different parts of the organisation work together, and how compliance will be checked. The policy must also align with all relevant laws, directives, regulations, and internal standards.

The chief information security officer (CISO), or a delegated policy owner, must be formally responsible for producing, documenting, and distributing the policy to the CISO, the configuration management board, system owners, security and privacy programme leads, and IT operations leadership. The CISO (or delegated policy owner) must also distribute the procedures to the configuration management team, change managers, system administrators, operational technology (OT) engineering leads, and security and privacy programme leads.

The policy must be reviewed and updated at least once every year, and after any event that requires review. The procedures must be reviewed and updated at least once every quarter, and after audit findings, security incidents or breaches, major legal or regulatory changes, or material changes to the system architecture.

### CM-2 — Baseline Configuration (Control)

This requirement is about keeping an agreed, up-to-date “starting point” for how your system is set up. This includes how it connects to other systems, how it works, how its parts share information, and what protections are in place to keep data safe. Without this, teams may unintentionally change settings or connections, which can leave protection gaps, cause outages, or make it difficult to explain what was in place after an incident.

To meet this expectation, the organisation must create and document the system’s current baseline, and keep it up to date using formal change control, so that updates are approved and recorded. The baseline must be reviewed and updated at least every quarter, and also whenever there are significant changes to the system’s design, its protection measures, its network connections, or the ways it communicates. This must be done after any major incident that affects the configuration, and when any system component is installed or upgraded.

#### CM-2.3 — Retention of Previous Configurations (Enhancement)

This requirement is about keeping a safe, retrievable record of the last three approved “starting settings” for your systems. This lets you quickly restore the system to a previously approved state if a change causes problems. Without this, an error, a faulty update, or an unintended change could prevent you from restoring an earlier approved configuration, leading to longer outages, more disruption, and a higher risk of operating in an unsafe or non-compliant way.

To meet this expectation, the organisation must keep **exactly the last three approved baseline configuration versions**, including all information needed to restore them. This includes hardware, software, and firmware details, the configuration files and records, and the supporting documentation.

The organisation must also have clear written rules, day-to-day procedures, and a plan that explain how these versions are created, approved, stored, and kept for rollback. Named people must be responsible for managing and overseeing this work. There must also be evidence that the retained versions can be retrieved and used during a controlled restore.

Without this separation, changes made while building or testing new features could accidentally carry over into the live environment. To meet this expectation, the organisation must keep documented baseline settings for both development and test environments. Each baseline must be kept distinct from the live baseline. Where practicable, test setups should be made to resemble the live setup so that results are realistic, even if the separation is achieved through management processes rather than separate physical machines.

#### CM-2.7 — Configure Systems and Components for High-risk Areas (Enhancement)

This requirement is about making sure that laptops and mobile devices used for travel are prepared to be as secure as possible before someone goes to a higher-risk location. After travel, the devices must be thoroughly checked and cleaned before they are used again. Without this, a device could be physically interfered with during travel, have harmful software added, or be changed in ways that allow sensitive information to leak or be accessed by the wrong people.

The organisation must first decide which travel destinations are considered to be significant risk. It must then provide only its managed travel laptops and mobile devices to staff travelling to those destinations. Before travel, the organisation must wipe the device storage, protect the device storage using encryption, install only a minimal set of approved applications, and set the device to the organisation’s more secure standard settings.

When staff return, the organisation must inspect the device for signs of tampering, run a malware (harmful software) check, wipe and reinstall a clean operating system, and then restore the device using approved, known-good system images before returning it to normal use.

### CM-3 — Configuration Change Control (Control)

This requirement is about making sure any change to your systems is properly planned, checked, and formally agreed before it is put into live use. Without this, staff might make unplanned or risky updates that could accidentally disrupt services, weaken protections, or expose personal information, including when urgent fixes are needed.

The organisation must clearly set out which types of system changes are always treated as “controlled”. It must also keep clear written procedures for how changes are proposed, explained, tested, checked, approved or rejected, carried out, and then formally closed off.

All controlled changes must be reviewed by the Change Advisory Board, which meets every week. Before it decides, the board must explicitly consider both security and privacy impacts. The organisation must record the board’s decision, implement only what was approved, and keep complete records for 365 days.

The organisation must also monitor and review what happens before and after each change, including emergency changes, changes that affect security or privacy, and changes made to fix known weaknesses.

#### CM-3.2 — Testing, Validation, and Documentation of Changes (Enhancement)

This expectation is about making sure every change to your systems is tested, checked, and properly recorded before it is allowed to go live. Without this, even a small update to hardware, software, firmware, or settings could cause services to fail, disrupt day-to-day operations, or create problems that are only found after customers or staff are affected. To meet this, the organisation must plan and carry out testing for each change, then confirm the results match what was expected, and keep clear evidence that this was done, before final approval.  

The change must also be fully documented, including what was changed, what it affects, and where the supporting records are stored. Testing should be arranged so it does not interfere with business needs. If the system must be taken offline, this should happen during planned outages. If testing on the live system is not possible, suitable additional safeguards must be used and documented.

### CM-7 — Least Functionality (Control)

This expectation is about keeping systems as simple as possible. Only the features you truly need should be available, and anything risky or unnecessary should be turned off or removed. Otherwise, attackers can take advantage of leftover “default” features, older or insecure ways of connecting, and hidden services to break in, steal data, or take over systems—especially through file sharing, anonymous access, guest logins, weak remote access methods, or unauthorised tools.

To meet this, the organisation must configure systems so they allow only:
- access by authenticated users,
- the required application services,
- approved management access, and
- necessary security monitoring.

It must prohibit or restrict:
- file sharing,
- directory browsing,
- anonymous access,
- dynamic DNS updates,
- guest accounts, and
- unused web administration.

It must also block specific network connection points and legacy methods, including Telnet, File Transfer Protocol (FTP), Trivial File Transfer Protocol (TFTP), Server Message Block version 1 (SMBv1), Remote Shell (RSH), and rlogin. Web access must use secure encrypted connections only. The organisation must remove or disable default sample applications, unused drivers, browser plug-ins, remote desktop agents, and unauthorised third-party software. It must disable anonymous web services and unnecessary discovery features.

Remote administration must be limited to approved source locations and must require a second factor, such as a code or token.

### CM-8 — System Component Inventory (Control)

### CM-14 — Signed Components (Control)

This requirement is about making sure only trusted software and device updates are allowed to be installed. Before any update—such as a patch, service pack, device driver, or BIOS/UEFI firmware update—is installed on your systems, the organisation must check that it has a digital signature and that this signature was created using a certificate the organisation recognises and has approved. Without this, attackers could disguise harmful or altered code as a legitimate update, which could allow them to take over systems, steal data, or disrupt operations.

To meet this expectation, the organisation must:
- keep a clear list of approved signing certificates
- define which software and firmware items must be signed
- ensure the installation process blocks anything that is not signed, or that has a signature that does not pass verification

The organisation must also:
- document who is responsible for checking certificates
- include this approach in its configuration management and system security planning
- keep records of authorised changes and maintain an inventory of components
- use automated installation or checking tools where available to enforce the rule for both IT systems software and operational technology (OT) device firmware
